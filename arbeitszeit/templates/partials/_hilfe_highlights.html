Sensible Dokumente & Schluesselverwaltung
Dokumente: AES-128 Fernet fuer Zeugnisse, Abschluesse, Reiseunterlagen. Kein Klartext in der Datenbank. Unveraenderbares Zugriffsprotokoll mit IP, User, Zeitstempel.
Private Signatur-Schluessel (neu): Jeder Mitarbeiter-Schluessel wird mit PBKDF2-HMAC-SHA256 (600.000 Iterationen) aus dem Login-Passwort abgeleitet und mit AES-256-GCM verschluesselt gespeichert. Der Entschluesselungsschluessel existiert nur waehrend der aktiven Sitzung im Arbeitsspeicher – nie auf der Festplatte oder in der Datenbank.
Einordnung: Ein gestohlener Datenbank-Dump allein reicht nicht aus, um Signaturen zu faelschen. Dafuer waere zusaetzlich eine aktive Nutzersitzung notwendig.

Was ist das ueberhaupt?

Stell dir vor, du nutzt WhatsApp zum Chatten und Zoom fuer Videokonferenzen — beides laeuft aber auf einem Server in eurer eigenen Firma. Kein amerikanisches Unternehmen sieht deine Nachrichten, kein Cloud-Anbieter speichert deine Gespraeche, keine Lizenzkosten, keine AGBs. Genau das ist Matrix (Chat) und Jitsi (Video).

Matrix / Element – der Firmen-Chat

Raeume sind das Herzstuck. Ein Raum ist wie eine WhatsApp-Gruppe, nur besser:

  • #allgemein – alle Mitarbeiter
  • #facility-elektro – nur das Elektro-Team
  • #hr-intern – nur HR, extra verschluesselt
  • #raum-101 – automatisch erstellt bei Raumbuchung
  • Direktnachrichten – 1:1, komplett privat

Du wirst automatisch in die richtigen Raeume eingeladen. Wenn ein Mitarbeiter ausscheidet: Account gesperrt, aus allen Raeumen entfernt — sofort, nicht erst wenn jemand daran denkt die WhatsApp-Gruppe zu bereinigen.

Jitsi Meet – die Videokonferenz

Wie Zoom — aber auf eurem eigenen Server. Keine Lizenz, keine Daten in amerikanischen Rechenzentren.

Aus der Raumbuchung: Raum in PRIMA buchen → im Buchungsdetail erscheint automatisch ein Button „Meeting beitreten“. Link per Matrix an Teilnehmer schicken — fertig. Externe Gaeste koennen per Link beitreten, kein Account noetig.

Verschluesselung – drei Ebenen erklaert

Stell dir einen Brief vor. Es gibt drei Wege ihn zu schuetzen:

Ebene 1 – Der Umschlag (TLS)
Der Brief steckt im Umschlag. Niemand auf dem Postweg liest ihn. Der Server selbst koennte ihn aber oeffnen. → Standard auf allen Webseiten (https://).
Ebene 2 – Tresor auf dem Server (Verschluesselung at Rest)
Nachrichten werden verschluesselt gespeichert. Wer die Festplatte stiehlt, liest nichts. → Matrix macht das automatisch.
Ebene 3 – Tresor in deinem Geraet (E2EE)
Die Nachricht wird auf deinem Geraet verschluesselt, bevor sie den Server erreicht. Der Server sieht nur unlesbaren Datenmuell. Nur das Geraet des Empfaengers kann entschluesseln. → Matrix/Element standardmaessig in privaten Raeumen. → Jitsi optional bei sensiblen Gespraechen.

Bei Matrix/Jitsi auf eurem Server musst du nicht vertrauen — technisch ist es schlicht unmoeglich fuer Aussenstehende mitzulesen, weil der Schluessel nur auf euren Geraeten liegt.

Was PRIMA automatisch macht

  • Raum buchen → Jitsi-Link + Matrix-Raum-Link sofort im Buchungsdetail
  • Stoermeldung abschicken → Matrix-Ping ans Facility-Team
  • Geplant: Neuer Mitarbeiter → Matrix-Account automatisch, direkt in Teamraeume eingeladen
  • Geplant: Mitarbeiter tritt aus → ueberall sofort gesperrt

E2EE bei Videokonferenzen – wann einschalten?

  • Standard (ohne E2EE): Verbindung zum Jitsi-Server verschluesselt (TLS). Da der Server euch gehoert: kein Problem.
  • Mit E2EE: Bild und Ton werden auf deinem Geraet verschluesselt. Server sieht nur Datenpakete. Empfohlen fuer Personalgespräche, GF-Sitzungen, Verhandlungen.
  • Voraussetzung E2EE: Chromium-basierter Browser (Edge, Chrome) oder Jitsi-App – Firefox noch nicht vollstaendig unterstuetzt.
Kurz zusammengefasst:
Matrix/Element ist euer WhatsApp – auf eurem Server, niemand liest mit.
Jitsi ist euer Zoom – kostenlos, ohne Lizenzvertrag, mit echter E2EE fuer sensible Gespraeche.
PRIMA verbindet alles: Raum buchen → Meeting-Link da. Stoerung melden → Team sofort benachrichtigt.

Technisch: Matrix verwendet Megolm (basiert auf Signal-Protokoll) fuer E2EE. Jitsi verwendet WebRTC Insertable Streams. Beide Verfahren gelten als sicher und sind Open Source – der Code ist oeffentlich pruefbar.

BACKUP-KONZEPT
Datensicherung im NAS/SAN-Umfeld
Warum PRIMA hier einen strukturellen Vorteil hat:
PRIMA nutzt kein Dateisystem — alle Dokumente, Metadaten, Protokolle und Binaerdaten stecken in einer einzigen PostgreSQL-Datenbank. Kein separates Dokumentenverzeichnis, kein verteilter Speicher, kein Sync-Problem. Ein einziger pg_dump sichert den vollstaendigen Systemzustand.

Ausbaustufe Technologie Werkzeug Recovery Time
Einstieg TrueNAS Scale (ZFS, iSCSI) pgBackRest + ZFS-Snapshot ~30 Min
Standard Synology RS-Serie + Proxmox Backup Server WAL-G + S3/MinIO ~10 Min
Professional NetApp ONTAP / Dell EMC PowerStore pgBackRest + Storage-Snapshot < 2 Min
High Availability Ceph / PostgreSQL Streaming Replication WAL-Archivierung + Standby-Server < 30 Sek (Failover)
Empfohlener Stack fuer Selbstbetrieb (IT-Standard):

pgBackRest
PostgreSQL-nativer Backup-Standard. Inkrementell, komprimiert, verschluesselt. WAL-Archivierung fuer Point-in-Time-Recovery — Datenbank auf beliebigen Zeitpunkt innerhalb des Archivfensters zuruecksetzen.
WAL-G
Schreibt PostgreSQL WAL-Segmente direkt auf S3-kompatible Ziele (MinIO, Ceph, NetApp S3). Backup-Overhead nahe null, da parallel zum laufenden Betrieb. Von Yelp entwickelt, heute Industrie-Standard.
TrueNAS Scale
Open-Source-NAS auf ZFS-Basis. Snapshots in Millisekunden, Copy-on-Write, integriertes iSCSI-SAN, S3-Gateway, Replikation auf zweites TrueNAS. IT-Community-Favorit fuer Heimlabore bis mittleres Rechenzentrum.
Proxmox Backup Server
Open-Source-Backup fuer VMs und Container, aber auch fuer Daten-Backups nutzbar. Deduplizierung, Verschluesselung, Web-GUI, Retention-Policies. Ideal wenn PRIMA in einer VM laeuft.
MinIO
S3-kompatibler Objektspeicher, laeuft auf eigener Hardware. WAL-G und pgBackRest sprechen MinIO nativ an — gleiche API wie AWS S3, keine Cloud-Abhaengigkeit. Cluster-faehig ab 4 Nodes (Erasure Coding).
Restic + Restic REST Server
Modernes Backup-Werkzeug: dedupliziert, verschluesselt (AES-256-CTR), laeuft gegen NAS, S3, SFTP oder lokales Verzeichnis. Einfachste Option fuer den Einstieg — ein Binary, kein Daemon.
3-2-1-Regel in der Praxis mit PRIMA:

3 Kopien:
  [1] Live-PostgreSQL        → PRIMA laeuft
  [2] TrueNAS Scale NAS     → taeglich, pgBackRest, 30 Tage Retention
  [3] Zweiter Standort / Band → woechentlich, verschluesselt, 1 Jahr

2 verschiedene Medien:
  NVMe (DB-Server) ≠ HDD/SSD-Pool (TrueNAS)

1 raeumlich getrennte Kopie:
  Anderes Gebaeude, anderer Brandabschnitt, oder
  verschluesselt auf Band im Safe (LTO-8: 12 TB nativ)
Point-in-Time-Recovery — was das bedeutet:
Mit WAL-Archivierung kann PRIMA auf jeden beliebigen Zeitpunkt der letzten N Tage zurueckgesetzt werden — nicht nur auf den letzten Backup-Punkt. Beispiel: Ransomware-Angriff Dienstag 14:32 Uhr → Restore auf Dienstag 14:31 Uhr. Datenverlust: eine Minute.

BSI CON.3 Pflicht: Backup ohne dokumentierten Restore-Test gilt als nicht vorhanden. Mindestens einmal jaehrlich vollstaendigen Restore auf Testsystem durchfuehren und protokollieren. PRIMA-Roadmap: Datensicherungs-GUI mit Protokoll und Faelligkeitserinnerung.

SKALIERUNGSSTRATEGIE
Enterprise-Skalierung: MinIO & OpenSearch
PRIMA ist von Grund auf fuer lineares Wachstum ausgelegt. Fuer Volumina im sechsstelligen Dokumentenbereich pro Jahr stehen zwei battle-tested Infrastrukturkomponenten bereit, die nahtlos in den bestehenden Stack integriert werden:

MinIO – S3-kompatibles Objekt-Storage
Ab einem bestimmten Dokumentenvolumen ist es technisch sinnvoll, Binaerdaten aus PostgreSQL in einen dedizierten Objekt-Store auszulagern. MinIO (Apache 2.0) bietet S3-kompatibles Storage auf eigener Hardware – ohne Cloud-Abhaengigkeit, ohne laufende Lizenzkosten, mit nativem Erasure Coding fuer Ausfallsicherheit. PostgreSQL behaelt dann nur Metadaten und Suchvektoren (~1 KB/Dokument), waehrend MinIO die Rohdateien verwaltet. Das Ergebnis: PostgreSQL bleibt in jedem Szenario schnell; Speicherkosten sinken auf Commodity-Niveau; horizontale Skalierung des Storage-Tiers wird moeglich.

OpenSearch – verteilte Volltextsuche
PostgreSQL FTS mit GIN-Index skaliert zuverlaessig bis in den mittleren einstelligen Millionenbereich. Sollte das Suchvolumen – durch steigende Gleichzeitigkeit, mehrsprachige Inhalte oder Faceted-Search-Anforderungen – ueber diesen Schwellwert hinauswachsen, ist OpenSearch (Apache 2.0, AWS-gepflegt) die logische Erweiterung. Die API ist kompatibel zu Elasticsearch 7.x; ein Django-Client existiert; der Indexierungsaufruf in PRIMA ist bereits als separater Service-Layer isoliert und laesst sich ohne Strukturaenderungen auf einen externen Search-Cluster umlenken.

Einordnung: Beide Komponenten sind fuer PRIMA vorbereitet, aber noch nicht aktiv – weil sie bei aktuellem Volumen schlicht nicht noetig sind. PostgreSQL FTS auf GIN-Index beantwortet Suchanfragen ueber 500 000 Dokumente in unter 50 ms. Das ist der richtige Zeitpunkt um zu skalieren: wenn die Messwerte es erfordern, nicht wenn ein Whitepaper es empfiehlt.

GESAMTBEWERTUNG
Datenhoheit & Selbstbetrieb
PRIMA wurde von Anfang an fuer genau ein Szenario gebaut: vollstaendig selbst betrieben, ohne Abhaengigkeit von Drittanbietern. Kein CDN, keine externen APIs, keine Cloud-Pflicht. Alle Daten bleiben dort wo sie hingehoeren — im eigenen Haus.

Auf eigener Serverhardware mit lokaler PostgreSQL-Instanz ist PRIMA produktionsreifer als auf jedem Cloudanbieter: die Datenbanklatenz faellt von ~15 ms auf unter 1 ms, der OnlyOffice-Callback laeuft im eigenen LAN, kein Container schlaeft ein, und die Rechenleistung ist dediziert statt geteilt.

Railway und Supabase waren immer nur ein Kompromiss fuer die Entwicklungsphase. Das Ziel war immer der eigene Serverraum.

Dieselbe Arbeit faellt bei SAP, Personio oder Microsoft 365 genauso an — dort zahlt man dafuer sechsstellige Jahresbetraege und bekommt im Gegenzug Daten auf fremden Servern, Vendor Lock-in und Abhaengigkeit von Kuendigungsfristen.

Fazit: Wer PRIMA auf eigener Hardware betreibt, hat ein vollstaendiges Unternehmenssystem das ihm gehoert, das er versteht, das er anpassen kann — und das keine Daten nach aussen traegt. Das ist keine Bastelarbeit. Das ist Infrastruktur.

HIGHLIGHT
Sicherheits-App: Brand- & AMOK-Alarmierung
Vollstaendig digitalisierter Gefahrenalarm-Ablauf — direkt ins Intranet integriert, ohne externe App, ohne Cloud. Bei Brandalarm loest PRIMA automatisch eine gestaffelte Erkunder-Alarmierung per Matrix-DM aus: naechste Etage zuerst, dann stufenweise weiter. Erkunder melden sich per Token-Link zurueck (kein Login noetig). Security-Personal sieht die Einsatzleitstelle in Echtzeit und entscheidet ueber Vollalarm oder Entwarnung.
Brand-Ablauf: Melder → PRIMA startet Alarm → Matrix-DM an alle Erkunder (gestaffelt nach Etage) → Erkunder antwortet per Token (1=Unterwegs, 2=Vor Ort, 9=Abbruch) → Einsatzleitstelle (Security) → Security-Review → Vollalarm oder Entwarnung

KONZEPT
📡 TETRA-BOS-Anbindung
PRIMA kann den EH-Alarm direkt an TETRA-BOS-Digitalfunkgeräte senden — und Erst­helfer können per einzigem Tastendruck am Funkgerät rückmelden (Status 3 = Unterwegs, Status 4 = Vor Ort), ohne Smartphone, ohne Login, ohne Link öffnen. Das Konzept ist technisch vollständig ausgearbeitet und in einem Arbeitstag implementierbar, sobald ein TETRA- oder DMR-Gateway im Netz erreichbar ist. Bidirektionale Kopplung: PRIMA sendet SDS-Textnachricht → Ersthelfer drückt Statustaste → PRIMA empfängt Webhook → Protokoll und Lauftext-Banner werden identisch wie bei Matrix/ntfy aktualisiert.
