1. Was ist TETRA-BOS?

TETRA (Terrestrial Trunked Radio) ist ein digitaler Bündelfunk-Standard, der in Deutschland als BOS-Digitalfunk (Behörden und Organisationen mit Sicherheitsaufgaben) betrieben wird. Das Netz wird von der BDBOS (Bundesanstalt für den Digitalfunk der BOS) verwaltet und deckt nahezu 100 % des deutschen Staatsgebiets ab — einschließlich Tunneln, Bahnhöfen und Gebäuden.

Kerneigenschaften TETRA
  • Verschlüsselte Sprach- & Datenkommunikation (TEA2)
  • Gruppenruf (ein Sender, viele Empfänger gleichzeitig)
  • Direktbetrieb (ohne Basisstation, Gerät zu Gerät)
  • Priorisiertes Notfallnetz — funktioniert auch bei Netzüberlast
  • Eingebauter GPS-Ortungsdienst
  • SDS (Short Data Service) — TETRA-eigenes Datennachrichtenformat
Nutzer im BOS-Netz
  • Polizei (Bund und Länder)
  • Feuerwehr (Berufs- und Freiwillige)
  • Rettungsdienst / Notarzt
  • THW, Katastrophenschutz
  • Werksfeuerwehren (mit BOS-Zulassung)
  • Betriebssanitätsdienste (je nach Bundesland)

2. Wer hat Zugang zum BOS-Netz?

Das BOS-TETRA-Netz ist kein offenes Netz. Der Zugang ist gesetzlich geregelt (BDBOS-Gesetz) und erfordert eine offizielle Zulassung.

Organisation BOS-Zugang Voraussetzung
Polizei, Feuerwehr, Rettungsdienst, THW Ja Gesetzlich zugewiesen
Werksfeuerwehr (z.B. Chemie, Flughafen, Bahn) Ja Anerkannte Werksfeuerwehr + Antrag beim Landesinnenministerium
Betriebssanitätsdienst (anerkannt) Eingeschränkt Variiert je Bundesland, Kooperation mit Rettungsdienst erforderlich
Privates Unternehmen ohne BOS-Status Nein Kein direkter Zugang — privates TETRA oder Alternativen nutzen
Zuständigkeit: Anträge auf BOS-Zulassung laufen über das jeweilige Landesinnenministerium und die BDBOS. Für Werksfeuerwehren ist zusätzlich die Brandschutzdienststelle des Landkreises einzubinden.

3. Bidirektionale Rückmeldung — genau wie Element

Das ist der native Anwendungsfall von TETRA — das Protokoll wurde von Grund auf für genau diesen Zweck gebaut: Alarm senden, Statusrückmeldung empfangen, ohne dass der Ersthelfer tippen oder einen Link öffnen muss.

Vorteil gegenüber Matrix/Element: Der Ersthelfer muss kein Smartphone herausholen, keinen Link öffnen, sich nicht einloggen. Er drückt am Funkgerät eine einzige Taste — PRIMA empfängt den Status sofort. Das funktioniert auch wenn das Mobilfunknetz überlastet oder ausgefallen ist.

4. Technische Architektur

PRIMA Django (Alarm auslösen) │ │ HTTP POST → Gateway-API │ Nachricht: "EH-ALARM Serverraum EG – 09:42 Uhr" │ Empfänger: ISSI-Gruppe "Ersthelfer" oder Einzel-ISSI ▼ TETRA-IP-Gateway (z.B. Frequentis 3020 IDC / Sepura SRG3900) │ │ SDS-4 Textnachricht (140 Zeichen) an Gruppe/Einzelgerät ▼ TETRA-Endgeräte der Ersthelfer │ Bildschirm zeigt: "EH-ALARM Serverraum EG – 09:42" │ Lautsprecher: Alarmton │ │ Ersthelfer drückt [Status 3 = Unterwegs] oder [Status 4 = Vor Ort] ▼ TETRA-IP-Gateway (empfängt SDS-Status) │ │ HTTP POST Webhook → /ersthelfe/tetra/rueckmeldung/ │ {"issi": "0912345", "status": 4} ▼ PRIMA Django (Rückmeldung verarbeiten) │ │ ISSI → HRMitarbeiter auflösen │ ErsteHilfeRueckmeldung anlegen (identisch zu Token-Rückmeldung) │ Matrix-Ping im EH_PING-Raum ▼ Vorfall-Protokoll aktualisiert – Lauftext-Banner erlischt
Die ISSI (Individual Short Subscriber Identity) ist die eindeutige Kennnummer jedes TETRA-Geräts — vergleichbar mit einer Mobilfunknummer. In den HR-Stammdaten des Ersthelfers (Modell HRMitarbeiter) wird die ISSI als neues Feld hinterlegt. PRIMA löst darüber den Mitarbeiter auf, legt die ErsteHilfeRueckmeldung an — dieselbe Logik wie beim Token-Link.

6. PRIMA-Integration im Detail

In PRIMA würde eine neue App tetra/ entstehen. Der Aufwand ist gering, weil die gesamte EH-Logik (Rückmeldeverarbeitung, Protokoll, Matrix-Ping) bereits vorhanden ist — TETRA ist nur ein weiterer Eingangskanal.

Neues Feld in HR-Stammdaten
# hr/models.py (Ergänzung HRMitarbeiter) tetra_issi = models.CharField( max_length=15, blank=True, default="", verbose_name="TETRA ISSI", help_text="Individuelle Teilnehmerkennung des TETRA-Funkgeraets" )
Webhook-Endpunkt (Rückmeldung empfangen)
# tetra/views.py @require_POST def tetra_rueckmeldung(request): """Empfaengt SDS-Statusmeldung vom TETRA-IP-Gateway.""" daten = json.loads(request.body) issi = daten.get("issi", "") status = int(daten.get("status", 0)) # ISSI → Mitarbeiter auflösen STATUS_MAP = {3: "unterwegs", 4: "am_ort", 5: "unterstuetzung", 6: "abgelehnt"} prima_status = STATUS_MAP.get(status) if not prima_status: return JsonResponse({"ok": False, "fehler": "Unbekannter Status"}, status=400) try: mitarbeiter = HRMitarbeiter.objects.get(tetra_issi=issi) except HRMitarbeiter.DoesNotExist: return JsonResponse({"ok": False, "fehler": "ISSI unbekannt"}, status=404) # Offenen Vorfall finden und Rückmeldung anlegen vorfall = ErsteHilfeVorfall.objects.filter(status="offen").order_by("-erstellt_am").first() if not vorfall: return JsonResponse({"ok": False, "fehler": "Kein offener Vorfall"}) ErsteHilfeRueckmeldung.objects.create( vorfall=vorfall, ersthelfer=mitarbeiter, status=prima_status ) return JsonResponse({"ok": True})
Alarm senden (SDS-4 Textnachricht)
# tetra/services.py def tetra_alarm_senden(vorfall): """Sendet SDS-4-Textnachricht an die TETRA-Ersthelfer-Gruppe.""" from django.conf import settings import urllib.request, json nachricht = "EH-ALARM {} – {} Uhr – Rueckmeldung per Status 3/4/5/6".format( vorfall.ort, vorfall.erstellt_am.strftime("%H:%M"), ) payload = { "destination": settings.TETRA_EH_GRUPPE, # z.B. Gruppen-GSSI "message": nachricht, "priority": 15, # TETRA-Priorität 0–15, 15 = höchste } req = urllib.request.Request( settings.TETRA_GATEWAY_URL + "/api/sds/send", data=json.dumps(payload).encode("utf-8"), headers={"Content-Type": "application/json", "Authorization": "Bearer " + settings.TETRA_GATEWAY_TOKEN}, method="POST", ) urllib.request.urlopen(req, timeout=5)
settings.py benötigt drei neue Variablen: TETRA_GATEWAY_URL, TETRA_GATEWAY_TOKEN, TETRA_EH_GRUPPE (GSSI der Ersthelfer-Gruppe). Alle aus .env.prima geladen — kein Hardcoding.

7. Alternativen ohne TETRA-Zulassung

Wenn kein Zugang zum BOS-TETRA-Netz besteht und kein privates TETRA aufgebaut werden soll, gibt es praxistaugliche Alternativen mit ähnlichen Eigenschaften:

DMR (Digital Mobile Radio)
Günstigerer digitaler Funk, weit verbreitet in Industrie und Gewerbe. Hytera, Motorola, Kenwood bieten DMR mit IP-Gateway. Statusrückmeldungen per Tastendruck — identische Logik wie TETRA SDS.
Hardware: Hytera IP SITE CONNECT Gateway (ab ca. 800 EUR), DMR-Handfunkgeräte ab ca. 200 EUR.
POCSAG-Pager
Klassische Alarmierungstechnik, die in vielen Feuerwehren parallel zu TETRA läuft. Nur Empfang (keine Rückmeldung per Gerät möglich). Günstigste Einstiegslösung für reine Alarmierung.
Hardware: Swissphone DE900, POCSAG-Sender für Intranet (z.B. über RaspberryPi + CC1101-Modul).
LTE-PTT (Push-to-Talk over LTE)
Nutzt das öffentliche Mobilfunknetz. Kein eigenes Funknetz nötig. Statusrückmeldungen per App oder Hardtaste möglich. Abhängig von Mobilfunkabdeckung — versagt bei Netzausfall.
Anbieter: Hytera PNC-Serie, Motorola WAVE PTX, Zello (für einfache Anwendungen).
Aktuell in PRIMA: ntfy + Matrix
Bereits implementiert und funktionsfähig. Kein zusätzliches Gerät. Abhängig von WLAN / Mobilfunknetz und Akku des Smartphones. Rückmeldung per Token-Link (4 Buttons).
Für die meisten Betriebe ausreichend — TETRA/DMR nur sinnvoll wenn erhöhte Ausfallsicherheit benötigt wird.

9. Implementierungsplan (wenn Hardware vorhanden)

Sobald ein TETRA- oder DMR-Gateway im Netz erreichbar ist, wäre der Implementierungsaufwand in PRIMA überschaubar — die EH-Kernlogik ist bereits vorhanden.

1
hr/models.py: Neues Feld tetra_issi bei HRMitarbeiter + Migration. Admin-Eintrag ergänzen.
2
config/settings.py: Drei neue Variablen aus .env: TETRA_GATEWAY_URL, TETRA_GATEWAY_TOKEN, TETRA_EH_GRUPPE.
3
neue App tetra/: services.py (Alarm senden), views.py (Webhook empfangen), urls.py.
4
ersthelfe/views.py: _benachrichtige_ersthelfer() um tetra_alarm_senden() erweitern — läuft parallel zu Matrix und ntfy.
5
Testen: Gateway simulieren mit curl -X POST /ersthelfe/tetra/rueckmeldung/ -d '{"issi":"0912345","status":4}'. ISSI muss in HR-Stammdaten eingetragen sein.
Geschätzter Aufwand: 1 Arbeitstag für PRIMA-Implementierung. Hardware-Beschaffung und Gateway-Konfiguration beim Hersteller sind der eigentliche Aufwand (1–3 Tage je nach Gateway-Typ und Hersteller-Support).

10. Fazit

Technisch: Vollständig lösbar
PRIMA kann TETRA bidirektional anbinden. Die gesamte Rückmeldungslogik (Vorfall, Protokoll, Matrix-Ping, Lauftext-Banner) ist bereits vorhanden und muss nur um einen neuen Eingangskanal erweitert werden. Aufwand PRIMA-seitig: 1 Tag.
Organisatorisch: Klärungsbedarf
BOS-TETRA erfordert offizielle Zulassung (Landesinnenministerium / BDBOS). Privates TETRA/DMR ist ohne Zulassung möglich, erfordert aber Frequenzlizenz (Bundesnetzagentur) und Gateway-Hardware.
Empfehlung: Für die meisten Betriebe ist das bestehende System (ntfy + Matrix/Element) ausreichend. TETRA/DMR ist sinnvoll wenn der Betrieb bereits eine Werksfeuerwehr, einen anerkannten Betriebsrettungsdienst oder eine Umgebung hat, in der Mobilfunk nicht zuverlässig verfügbar ist (Produktionshallen, Keller, Außengelände).
