1. Authentifizierung

Alle API-Anfragen benoetigen einen gueltigen Bearer-Token im HTTP-Header:

Authorization: Bearer <64-stelliger-Hex-Token>

Tokens werden im PRIMA-Adminbereich unter DMS → API-Tokens verwaltet. Jedes externe System erhaelt einen eigenen Token:

  • SAP ERP-System → eigener Token mit Berechtigung „Nur offen“
  • Paperless-ngx → eigener Token fuer Scan-Imports
  • Backup-System → eigener Token, jederzeit einzeln sperrbar

Token werden niemals im URL-Parameter übertragen — ausschließlich per HTTP-Header. Alle API-Anfragen sollten ausschließlich über HTTPS erfolgen.

2. Basis-URL & Format
Basis-URL {{ basis_url }}/dms/api/v1/
Antwortformat application/json (UTF-8)
Upload-Format multipart/form-data
Datum/Uhrzeit ISO 8601 mit Zeitzone, z.B. 2026-03-09T14:30:00+01:00
Zeichensatz UTF-8 durchgehend
Hinweis: Die API ist zustandslos — jede Anfrage muss den Authorization-Header enthalten. Cookies werden nicht ausgewertet.

Verbindungstest. Gibt an ob der Server erreichbar ist und welches System sich authentifiziert hat.

Neues Dokument hochladen (multipart/form-data).

Formular-Felder:

GET /dms/api/v1/dokumente/<id>/

Metadaten eines einzelnen Dokuments inkl. Versionsverlauf.

Antwort 200: Wie Dokumentenliste + Feld "versionen": [{"version_nr": 3, "erstellt_am": "...", "groesse_bytes": 204800, "kommentar": "..."}]
GET /dms/api/v1/dokumente/<id>/inhalt/

Rohen Dateiinhalt als Binär-Download. Content-Type entspricht dem MIME-Typ des Dokuments.

Antwort-Header: 
Content-Type: application/pdf
Content-Disposition: attachment; filename="brandschutz_2026.pdf"
X-PRIMA-Dokument-ID: 17
X-PRIMA-Version: 3
POST /dms/api/v1/dokumente/<id>/version/

Neue Version eines bestehenden Dokuments hochladen. Felder: datei (Pflicht) + kommentar (optional).

Antwort 201: 
{ "dokument_id": 17, "neue_version": 4 }
4. Fehlercodes

Fehlerantworten folgen immer dem Format: {"fehler": "...", "code": "FEHLERCODE"}

HTTP-StatusCodeBedeutung
401UNAUTHORIZEDToken fehlt oder ungueltig
403FORBIDDENToken hat keine Berechtigung fuer diese Aktion (z.B. sensible Dokumente)
404NOT_FOUNDDokument oder Kategorie nicht gefunden
400BAD_REQUESTPflichtfeld fehlt oder ungueltige Parameter
405Method Not AllowedHTTP-Methode wird fuer diesen Endpunkt nicht unterstuetzt
500—Serverseitiger Fehler (im PRIMA-Log protokolliert)
5. cURL-Beispiele

Verbindungstest:

curl -s \
  -H "Authorization: Bearer <TOKEN>" \
  {{ basis_url }}/dms/api/v1/health/

Dokument hochladen:

curl -s -X POST \
  -H "Authorization: Bearer <TOKEN>" \
  -F "datei=@/pfad/zum/dokument.pdf" \
  -F "titel=Lieferschein 2026-0042" \
  -F "klasse=offen" \
  -F "kategorie=3" \
  -F "beschreibung=Import aus SAP MM" \
  {{ basis_url }}/dms/api/v1/dokumente/

Dokumentliste abrufen (gefiltert):

curl -s \
  -H "Authorization: Bearer <TOKEN>" \
  "{{ basis_url }}/dms/api/v1/dokumente/?klasse=offen&suche=Lieferschein&limit=10"

Dateiinhalt herunterladen:

curl -s \
  -H "Authorization: Bearer <TOKEN>" \
  -o ausgabe.pdf \
  {{ basis_url }}/dms/api/v1/dokumente/17/inhalt/
6. SAP ABAP-Beispiel

Der folgende ABAP-Code nutzt CL_HTTP_CLIENT um ein Dokument aus SAP ins PRIMA-DMS zu uebertragen. Voraussetzung: PRIMA-Server ist als HTTP-Destination in SAP SM59 eingetragen.

SM59 → Neue RFC-Destination → Typ G (HTTP-Verbindung zu externem Server) → Host: {{ basis_url|slice:"7:" }}, Pfad-Prefix: /dms/api/v1/ 

REPORT z_prima_dms_upload.

* Konfiguration
CONSTANTS: c_destination TYPE rfcdest VALUE 'PRIMA_DMS',
           c_token       TYPE string   VALUE '<64-STELLIGER-HEX-TOKEN>'.

* Dateiinhalt (Beispiel: internes Byte-Feld)
DATA: lv_inhalt   TYPE xstring,
      lv_dateiname TYPE string VALUE 'lieferschein_0042.pdf',
      lv_titel     TYPE string VALUE 'Lieferschein 2026-0042'.

* HTTP-Client erzeugen
DATA: lo_client   TYPE REF TO if_http_client,
      lv_subrc    TYPE sy-subrc.

cl_http_client=>create_by_destination(
  EXPORTING destination = c_destination
  IMPORTING client      = lo_client
  EXCEPTIONS OTHERS     = 4 ).

IF sy-subrc <> 0.
  MESSAGE 'HTTP-Client konnte nicht erstellt werden' TYPE 'E'.
  RETURN.
ENDIF.

* Methode und Pfad setzen
lo_client->request->set_method( 'POST' ).
lo_client->request->set_header_field(
  name  = 'Authorization'
  value = |Bearer { c_token }| ).

* Multipart-Body aufbauen
DATA: lo_entity TYPE REF TO if_http_entity.
lo_client->request->if_http_entity~set_content_type(
  'multipart/form-data; boundary=prima_boundary' ).

lo_entity = lo_client->request->get_form_field( 'titel' ).
lo_entity->set_cdata( lv_titel ).

lo_entity = lo_client->request->get_form_field( 'klasse' ).
lo_entity->set_cdata( 'offen' ).

lo_entity = lo_client->request->get_form_field( 'datei' ).
lo_entity->set_data( lv_inhalt ).
lo_entity->set_content_type( 'application/pdf' ).
lo_entity->set_header_field(
  name  = 'Content-Disposition'
  value = |form-data; name="datei"; filename="{ lv_dateiname }"| ).

* Anfrage senden
lo_client->send(
  EXCEPTIONS http_communication_failure = 1 OTHERS = 2 ).

IF sy-subrc <> 0.
  MESSAGE 'Fehler beim Senden der Anfrage' TYPE 'E'.
  RETURN.
ENDIF.

* Antwort empfangen
lo_client->receive(
  EXCEPTIONS http_communication_failure = 1 OTHERS = 2 ).

DATA: lv_status   TYPE i,
      lv_antwort  TYPE string.
lo_client->response->get_status( IMPORTING code = lv_status ).
lv_antwort = lo_client->response->get_cdata( ).

IF lv_status = 201.
  MESSAGE |Dokument erfolgreich hochgeladen: { lv_antwort }| TYPE 'S'.
ELSE.
  MESSAGE |Fehler { lv_status }: { lv_antwort }| TYPE 'E'.
ENDIF.

lo_client->close( ).
Hinweis fuer SAP-Integration Suite (CPI): Bei Verwendung von SAP Integration Suite (frueherer SAP HCI/CPI) kann die HTTP-Kommunikation ueber einen iFlow mit HTTP-Adapter abgebildet werden. Der Bearer-Token wird als Header-Property konfiguriert. Kontaktiere den PRIMA-Administrator fuer den Token.
7. Sicherheitshinweise
  • HTTPS erzwingen: Alle API-Anfragen müssen über HTTPS laufen. Im Intranet-Betrieb über interne CA mit Nginx als TLS-Termination.
  • Token schmal halten: Jedes System erhält nur die minimal notwendigen Berechtigungen. SAP MM benötigt nur Nur offen, niemals Offen + Sensibel.
  • Token-Rotation: Tokens sollten regelmäßig erneuert werden. Im Admin alten Token deaktivieren → neuen Token erstellen → im SAP hinterlegen.
  • Audit-Trail: Jeder API-Zugriff (Upload und Download) wird im ZugriffsProtokoll mit Token-Bezeichnung und IP-Adresse protokolliert. Nicht löschbar (DSGVO Art. 5 Abs. 2).
  • Sensible Dokumente: Klasse-2-Dokumente (Personalakten etc.) sind standardmäßig für API-Tokens gesperrt. Zugriff nur nach expliziter Freigabe durch PRIMA-Administrator.
  • Netzwerk-Ebene: Im Selbstbetrieb zusätzlich Firewall-Regel: API-Endpunkte nur aus dem SAP-Applikationsserver-Subnetz erreichbar.
